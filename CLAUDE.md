# PetReview 프로젝트 지침

## 프로젝트 개요

* 반려동물 리뷰 플랫폼 (MVP 단계)
* 프론트엔드: GitHub Pages (정적 호스팅)
* 백엔드 DB: Supabase
* 백엔드 프록시: Vercel (API 키 보호용)

## 아키텍처 핵심 규칙

* 프론트는 순수 정적 파일 (HTML/CSS/JS) — 빌드 툴 없음
* Supabase 직접 호출은 프록시 서버를 통해서만 할 것
* 민감한 키(Supabase anon key, service key 등)는 절대 프론트 코드에 하드코딩 금지
* 환경변수는 Vercel 대시보드에서 관리

## 파일 구조

```
/
├── index.html
├── assets/
│   ├── css/
│   └── js/
├── api/          ← 프록시 서버 코드 (Railway/Vercel)
└── CLAUDE.md
```

## 코딩 규칙

* JS는 바닐라 JS 기본, 프레임워크 추가 시 먼저 확인 요청할 것
* CSS는 별도 파일로 분리 유지
* 함수/변수명은 영어, 주석은 한국어로 작성
* 코드 수정 시 기존 로직 함부로 삭제하지 말 것 — 변경 전 반드시 설명 먼저

## 작업 방식

* 큰 변경 전에는 무엇을 왜 바꾸는지 먼저 설명하고 진행
* 파일 여러 개 동시에 수정할 때는 어떤 파일을 건드리는지 목록 먼저 제시
* 모르는 부분 있으면 추측으로 진행하지 말고 질문할 것
* 작업 완료 후 테스트 방법도 함께 안내

## 작업자(빈느님) 협업 원칙

* 모든 작업은 main 브랜치까지 push 완료할 것 (커밋만 하고 끝내지 말 것)
* Supabase SQL 작업은 토큰이 있으니 직접 실행해서 진행할 것
* 빈느님 의견에 무조건 동의하거나 칭찬하지 말 것 — 장점과 단점을 냉정하게 분석하고 이유와 함께 설명할 것
* 더 나은 아이디어가 있으면 항상 먼저 제시할 것
* 위 원칙들은 매 작업마다 반드시 준수할 것

## 주의사항

* GitHub Pages는 서버사이드 코드 실행 불가 — 백엔드 로직은 반드시 프록시로
* Supabase 스키마 변경은 신중하게 — 변경 전 현재 구조 확인 먼저
* 모든 작업은 main 브랜치에 직접 push (GitHub Pages 배포 기준 브랜치)
